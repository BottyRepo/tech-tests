## Питання 1 — JavaScript / Event Loop

Є два фрагменти коду. Кожна функція виконує **математичні обчислення** (без читання з диску, без мережевих запитів) і займає **~10 секунд** виконання.

```js
function heavyCalc1() {
  return new Promise((resolve) => {
    let result = 0;
    const end = Date.now() + 10_000;
    while (Date.now() < end) {
      result += Math.sqrt(Math.random());
    }
    resolve(result);
  });
}

function heavyCalc2() {
  return new Promise((resolve) => {
    let result = 0;
    const end = Date.now() + 10_000; 
    while (Date.now() < end) {
      result += Math.sqrt(Math.random());
    }
    resolve(result);
  });
}

const [r1, r2] = await Promise.all([heavyCalc1(), heavyCalc2()]);
```

### Питання до кандидата:

1. Скільки секунд займе виконання `Promise.all` — **~10 сек чи ~20 сек**? Чому?
2. Чи виконуються ці дві функції **паралельно**? Поясни механізм.
3. Що таке **event loop** і як він пов'язаний з цією поведінкою?
4. Які методи оптимізації **швидкості** ти б застосував до цього коду?

## Питання 6 — Back-End (Node.js / TypeORM) / Concurrency та Race Conditions

Є сервіс для роботи з гаманцем користувача. Користувач викликає метод `withdraw`, щоб зняти кошти.

```typescript
@Injectable()
export class WalletService {
  constructor(
    @InjectRepository(Wallet)
    private walletRepo: Repository<Wallet>,
  ) {}

  async withdraw(userId: string, amount: number) {
    const wallet = await this.walletRepo.findOne({ where: { userId } });
    
    if (wallet.balance < amount) {
      throw new BadRequestException('Not enough funds');
    }

    wallet.balance -= amount;
    await this.walletRepo.save(wallet);

    return wallet;
  }
}
```

Уявімо, що користувач має баланс 100$. Через повільний інтернет або зловмисний скрипт, він відправляє **два паралельних HTTP-запити** на зняття по 100$. Обидва запити надходять у NestJS практично мілісекунда в мілісекунду.

### Питання до кандидата:

1. Що відбудеться з балансом в результаті виконання цих двох запитів? Поясни покроково (з огляду на те, що Node.js працює в одному потоці — чому це взагалі проблема?).
2. Запропонуй як мінімум **два різні підходи** до вирішення цієї проблеми на рівні бази даних або коду (в контексті SQL / TypeORM). Які плюси та мінуси у кожного з підходів?
3. Якщо цей бекенд розгорнутий в **Kubernetes на 10 подів** (інстансів `WalletService`), чи будуть працювати твої попередні рішення? Якщо ні, що потрібно змінити в архітектурі (наприклад, розподілені блокування, Redis, черги), щоб уникнути race conditions?

## Питання 7 — Back-End (Node.js) / Робота з великими даними

Є метод в NestJS сервісі, який запускається щоночі по крону і має витягнути користувачів з бази даних та відправити їм сповіщення через зовнішнє API:

```typescript
@Cron('0 0 * * *')
async sendNightlyEmails() {
  // Витягуємо всіх активних користувачів
  const users = await this.userRepository.find({ where: { isActive: true } }); 
  
  // Відправляємо листи
  await Promise.all(
    users.map(user => this.emailApiService.send(user.email, 'Nightly Update'))
  );
}
```

В системі наразі зареєстровано **1 500 000** активних користувачів.

### Питання до кандидата:

1. Назви як мінімум 3 критичні (fatal) проблеми в цьому коді, через які він гарантовано "покладе" сервіс або базу даних на продакшені.
2. Як би ти переписав процес читання 1.5 млн записів з бази даних, щоб не викликати `Out of Memory` (OOM)?
3. Як би ти оптимізував відправку у зовнішнє API, щоб не потрапити під Rate Limit (429 Too Many Requests) і при цьому не чекати виконання днями? 
4. Що станеться, якщо посеред виконання (наприклад, на 500,000-му юзері) под впаде або перезавантажиться? Як архітектурно зробити цей процес надійним, ідемпотентним і стійким до падінь?
